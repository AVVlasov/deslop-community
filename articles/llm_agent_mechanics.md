# 5 механик агентной разработки: методичка

Руководство по организации работы с LLM-агентами в разработке.  
Цель — выстроить систему, в которой агент получает нужный контекст в нужный момент, а не через ручной ввод одних и тех же инструкций в каждую сессию.

материал подготовлен сообществом [DeSlop](https://t.me/ai_deslop)

## 1. Instruction file (`Agent.md` / `CLAUDE.md`)

### Концепция

Instruction file — это markdown-файл, который агент автоматически включает в **system prompt** при старте каждой сессии в данном рабочем пространстве или репозитории.  
`CLAUDE.md` и `Agent.md` — эквивалентные форматы одного паттерна: persistent agent instructions.

Это не замена промпту, это **фиксированная часть system prompt** на уровне проекта. Всё, что там описано, действует всегда — без явного упоминания в каждом запросе.

Что стоит фиксировать:

- **Технологические ограничения**
  - язык и версия runtime;
  - разрешённые пакетные менеджеры и зависимости;
  - архитектурные паттерны (REST vs gRPC, DI-контейнер и т.д.).
- **Политики безопасности**
  - операции, требующие явного подтверждения (schema migrations, prod deployments);
  - запрет прямого доступа к production-окружениям без review.
- **Соглашения**
  - code style и naming conventions;
  - структура commit messages;
  - формат генерируемых артефактов (тесты, документация, ADR).

Instruction file поддерживает **иерархию**: глобальный (уровень пользователя) → проектный (репозиторий) → локальный (поддиректория). Более конкретный уровень перекрывает вышестоящий.

### Когда использовать

Правило простое: если агент должен знать это **в каждой задаче без исключений** — это instruction file. Если «иногда» — см. Skills.

### Внедрение

1. Создайте `Agent.md` в корне репозитория.
2. Зафиксируйте 10–20 инвариантных правил: стек, запреты, обязательные практики.
3. Личные предпочтения (стиль ответов, уровень детализации) вынесите в персональный instruction file на уровне пользователя.
4. Версионируйте файл в git — это часть инфраструктуры проекта.

## 2. Dynamic instructions (Skills)

### Концепция

Skills — это паттерн **динамической загрузки инструкций** в context window агента.  
Каждый skill — отдельный файл с описанием сценария применения и детальными инструкциями выполнения.

Агент читает только **описания** (metadata) всех skills постоянно, но полное содержимое skill попадает в контекст лишь при совпадении с текущим запросом.  
Это снижает потребление context window и сохраняет инструкции актуальными без раздувания system prompt.

Примеры skills:

- Code review checklist — правила проверки PR по стандартам проекта.
- Deployment runbook — шаги, pre/post-checks, rollback-сценарии.
- ADR template — структура и требования к архитектурным решениям.
- Test generation spec — стиль тестов, naming conventions, coverage-требования.

Структура каждого skill:

```
/skills
  /code-review
    skill.md          # описание: когда использовать, входные данные, формат выхода
    checklist.md      # детальный чеклист
    examples/         # референсные примеры
```

### Когда использовать

Если одно и то же объяснение задачи повторяется в нескольких сессиях — это кандидат на skill.

### Внедрение

1. В течение 1–2 недель логируйте повторяющиеся инструкции, которые вы даёте агенту.
2. Для каждого паттерна создайте skill: описание сценария, шаги выполнения, формат выхода.
3. Сложите skills в директорию репозитория — они версионируются и шарятся через git в команде.
4. Привяжите skills к slash-командам (`/review`, `/deploy`) для явного вызова.

## 3. Multi-agent orchestration (Sub-agents)

### Концепция

Sub-agent — это **автономный агент**, запускаемый основным агентом для выполнения изолированной задачи.  
У каждого sub-agent свой context window, свои tool permissions и, при необходимости, своя модель.

Паттерн решает две проблемы:

- **Context pollution**: задачи, порождающие объёмный промежуточный вывод (анализ codebase, парсинг логов), не занимают context window основной сессии.
- **Principle of least privilege**: агент получает только те инструменты, которые нужны для его роли.

Типичные роли sub-agents:

| Роль | Задача | Tool permissions |
|---|---|---|
| Codebase explorer | Обходит файлы, строит архитектурную карту, возвращает summary | read-only filesystem |
| Code reviewer | Проверяет PR по чеклисту | read-only: code, git history |
| Documentation agent | Генерирует документацию, changelog, ADR | read: code; write: docs only |
| Debug agent | Анализирует логи, группирует ошибки, предлагает гипотезы | read-only: logs, observability |

Каждый sub-agent может иметь собственный instruction file (`Agent.md`) и набор skills. Основному агенту он возвращает структурированный output, а не весь промежуточный контекст.

### Когда использовать

Если задача читает десятки файлов или данных, порождает объёмный промежуточный результат, или логически является отдельной ролью — это sub-agent.

### Внедрение

1. Выделите задачи, которые занимают непропорционально много контекста: анализ legacy code, code review, генерация документации.
2. Для каждой роли определите:
   - tool permissions (read-only / ограниченный write / запрет shell);
   - собственный instruction file;
   - формат возвращаемого output.
3. Настройте модель под задачу: легковесная — для массового чтения/анализа, более мощная — для сложных reasoning-задач.

## 4. Lifecycle hooks

### Концепция

Lifecycle hooks — это **обработчики событий жизненного цикла агента**, выполняющиеся до или после определённых действий.  
Это не инструкции о том, как думать. Это детерминированные side effects, не зависящие от поведения модели.

Основные типы событий:

- `pre-tool-use` — перед вызовом инструмента (bash, file write, DB query).
- `post-tool-use` — после выполнения инструмента.
- `session-start` / `session-stop` — при старте и завершении сессии.
- `pre-agent-action` — перед любым действием агента.

Примеры использования:

- **Pre-hook для bash** — блокировка деструктивных команд (`rm -rf`, `DROP TABLE`, прямые изменения prod-конфигов); при обнаружении опасного паттерна возвращает non-zero exit code, команда не выполняется.
- **Post-hook для file write** — автоматический запуск `prettier`/`eslint`/`black` после каждой записи файла агентом.
- **Post-hook для изменения ключевых модулей** — автоматический запуск связанных тестов.
- **Audit log** — запись всех write-операций агента в журнал для последующего review.

Сложные решения («эта операция безопасна?») можно делегировать отдельной fast-inference модели внутри hook — она принимает бинарное решение: allow / deny / request human confirmation.

### Когда использовать

Если фраза «не забудь прогнать тесты», «не используй rm -rf» или «форматни файл» повторяется в ваших промптах — это hook, а не инструкция.

### Внедрение

1. Составьте список **обязательных** и **запрещённых** действий агента при работе с кодом и инфраструктурой.
2. Реализуйте каждое как hook на соответствующее событие.
3. Для сложной логики принятия решений используйте легковесную модель (fast inference) внутри hook.
4. Логируйте все срабатывания hooks — это часть audit trail.

## 5. Tool use и внешние интеграции (MCP и аналоги)

### Концепция

Tool use — это механизм, при котором агент вызывает внешние функции/API как часть reasoning loop: получает результат, включает его в context window и продолжает рассуждение.

**Model Context Protocol (MCP)** — открытый стандарт для подключения внешних сервисов как tools к LLM-агентам. Любой MCP-совместимый сервер становится набором typed tools, видимых агенту наравне с встроенными.

Что можно подключить:

- **Version control** (GitHub/GitLab) — чтение PR, issues, commit history, file tree.
- **Issue trackers** (Jira/YouTrack) — чтение и создание задач, связи, статусы.
- **Observability** (Sentry, Prometheus, ELK) — ошибки, метрики, логи.
- **Databases** — SQL/NoSQL запросы для аналитики и проверки гипотез (read-only по умолчанию).
- **Internal APIs** — корпоративные сервисы, конфиг-менеджеры, service registry.

Tools доступны как в прямых сессиях, так и внутри skills и sub-agents.

### Принцип минимальных привилегий

По умолчанию — read-only. Write-операции (создание задач, комментариев, запуск пайплайнов) — только через sub-agents с явно ограниченным scope и, при необходимости, с human-in-the-loop подтверждением через hooks.

### Внедрение

1. Определите приоритетные системы для интеграции: git-хостинг, issue tracker, CI/CD, observability.
2. Подключите их через MCP или нативные tool integrations вашего агентного фреймворка.
3. Назначьте минимально необходимые права для каждого инструмента.
4. Включите инструменты в skills и sub-agents: агент должен сам ходить за данными, а не ждать, пока вы скопируете их в чат.

## Матрица выбора механики

| Вопрос | Механика | Примеры |
|---|---|---|
| Агент должен знать это в каждой задаче? | Instruction file (`agent.md`) | Стандарты стека, политики безопасности, code style |
| Нужно только при конкретном типе задачи? | Dynamic instructions (Skill) | PR checklist, deployment runbook, ADR template |
| Задача должна быть изолирована в отдельном context window? | Sub-agent | Codebase analysis, code reviewer, documentation agent |
| Должно выполняться автоматически по событию? | Lifecycle hook | Autoformat, lint, блокировка деструктивных команд |
| Нужны данные или действия во внешних системах? | Tool use / MCP | Git, issue tracker, БД, observability |

---

## Стоимость context window

Понимание того, что и когда попадает в context window, критично для производительности и стоимости:

| Механика | Нагрузка на context window |
|---|---|
| Instruction file | Высокая — присутствует в каждой сессии |
| Skills (metadata) | Низкая — только описание |
| Skills (full content) | Умеренная — только при активации |
| Sub-agent | Нулевая для основного агента |
| Lifecycle hooks | Нулевая — выполняются вне context |
| Tool use | Умеренная — только результат вызова |
